4  Metadata and Cataloguing

4.1 Chapter overview

A digitized photograph appears in a repository with the title “Old picture,” creator “Unknown,” date “1900s,” and no subject, place, rights, or provenance information. The image file opens correctly, but can a student discover it? Can a community member correct its description? Can staff distinguish the original photograph from the digital scan? Can another institution reuse the record? Can a future curator explain where the item came from and what may legally be done with it?

The central problem of this chapter is how to describe resources consistently enough for people and systems to identify, select, obtain, understand, manage, connect, and preserve them without erasing important context. Metadata makes collections legible. Cataloguing applies professional principles, standards, vocabularies, and judgment to create and maintain that metadata.

Metadata is not decoration added after digitization, nor is cataloguing simply typing information into fixed boxes. Every record is a structured claim about entities, names, subjects, relationships, provenance, rights, and access. Those claims affect discovery and memory. They may reproduce inherited bias, or they may make previously obscured people and relationships visible.

This chapter connects database structure from Chapter 3 to descriptive meaning. Students will distinguish metadata schemas from content standards, encoding formats, value vocabularies, and conceptual models; examine MARC 21, RDA, Dublin Core, IFLA LRM, and BIBFRAME; practice authority and subject work; and create a small, quality-checked metadata record for a local heritage object.

4.2 Learning objectives

After completing this chapter, you should be able to:

  1. Classify metadata as descriptive, administrative, structural, technical, rights, provenance, or preservation metadata.
  2. Distinguish conceptual models, element sets, content standards, value vocabularies, encoding formats, and application profiles.
  3. Create a Dublin Core record using defined elements, controlled values, identifiers, language tags, and rights information.
  4. Explain the purposes of authority control, identifiers, classification, and subject vocabularies.
  5. Evaluate and revise metadata for completeness, accuracy, consistency, provenance, accessibility, and interoperability.
  6. Analyze how descriptive choices affect representation, intellectual freedom, cultural rights, privacy, and discovery.

4.4 Core discussion

4.4.1 Metadata as service infrastructure

Metadata links users to resources. A catalogue result depends on recorded titles, names, subjects, identifiers, relationships, holdings, and access conditions. A repository browse depends on consistent departments, dates, resource types, and languages. A digital exhibit depends on captions, provenance, rights, and connections among objects. Preservation depends on file formats, checksums, events, agents, and dependencies.

The phrase “data about data” is useful but incomplete. Metadata may describe a physical object, a person, an organization, an event, a place, a right, or a relationship. In one context a city name is descriptive metadata; in a geographic authority file, the city is the primary entity. What matters is the statement being made: which property describes which entity, according to which rules, for which purpose.

NISO groups metadata broadly into descriptive, structural, and administrative functions (Understanding Metadata 2004). In practice, administrative metadata is often divided further. Table 4.2 shows how the categories support different work. One value can serve more than one function, so categories are analytical aids rather than sealed boxes.

As Table 4.2 shows, a digitized object needs more than a title. Its page order, file characteristics, rights, source, and preservation history may determine whether it remains usable and trustworthy.

4.4.2 A stack of models, rules, elements, values, and syntax

Students often say that MARC, RDA, and Dublin Core are competing “metadata standards.” They are related, but they do not do the same job. Figure 4.1 separates the layers.

Figure 4.1 is not a rigid production sequence. A project may begin with an existing platform or exchange format. The figure is a diagnostic tool: when records fail, ask whether the problem lies in the conceptual model, available elements, content rules, values, encoding, local profile, or maintenance.

A conceptual model defines entities and relationships at an abstract level. An element set defines properties such as title or creator. A content standard gives instructions for recording values. A value vocabulary supplies controlled names, subjects, types, or codes. An encoding format makes statements machine-processable. An application profile combines selected elements, obligations, vocabularies, and local rules for a specific community.

4.4.3 User tasks and bibliographic entities

Cataloguing is shaped by what users need to do. The IFLA Library Reference Model identifies broad user tasks: find, identify, select, obtain, and explore (IFLA Library Reference Model Review Group 2017). These tasks require both attributes and relationships. A user may need to distinguish a novel from its translation, identify an edition suitable for citation, obtain an accessible format, or explore other works by a creator.

IFLA LRM models bibliographic information through entities including Work, Expression, Manifestation, Item, Agent, Nomen, Place, and Time-span. In simplified terms:

  • Work concerns a distinct intellectual or artistic creation.
  • Expression concerns the realization of a work, such as a language or performance.
  • Manifestation concerns a publication or production embodiment.
  • Item concerns an individual exemplar.

These distinctions prevent common errors. A barcode identifies an item, while an ISBN generally identifies a manifestation. A translator contributes to an expression. Damage notes apply to an item. A publication statement belongs to a manifestation context.

Conceptual entities do not always appear as separate screens or database tables. They guide consistent description and help systems connect related resources.

4.4.4 MARC 21, RDA, Dublin Core, and BIBFRAME

Table 4.3 compares four frameworks students will encounter. The rows are deliberately phrased by role because the frameworks may be used together rather than selected as exact substitutes.

MARC 21 provides formats for communicating bibliographic, authority, holdings, classification, and community information. Its bibliographic format defines tags, indicators, subfields, and coded elements (Library of Congress 2024b). A MARC field such as 245 structures title information; cataloguing instructions determine which title to record and how.

RDA is designed around resource description and access, with entities, elements, relationships, and instructions. The open RDA Registry publishes element and vocabulary representations for machine use (RDA Steering Committee 2026). RDA is not a file format. RDA data may be encoded in MARC or other structures.

Dublin Core provides broadly applicable terms. The DCMI terms vocabulary includes properties such as title, creator, subject, date, type, format, identifier, source, language, relation, coverage, and rights (Dublin Core Metadata Initiative 2020). Its apparent simplicity is valuable for cross-domain projects, but projects must define obligations and value rules.

BIBFRAME is a Library of Congress initiative for representing bibliographic description as linked data. Its model centers on Work, Instance, and Item and uses RDF relationships (Library of Congress 2024a). Chapter 8 returns to RDF and linked data.

4.4.5 Elements, values, and application profiles

An element is not a screen label. It is a property with a definition, expected value, and domain of use. “Date” may mean creation, publication, capture, accession, modification, copyright, or availability. A profile that uses one undifferentiated date field will create ambiguity even if every value follows YYYY-MM-DD.

An application profile should document:

  • element name and URI where available;
  • definition and examples;
  • obligation: required, recommended, or optional;
  • repeatability;
  • expected value type or syntax;
  • controlled vocabulary or authority source;
  • language and script practices;
  • source and provenance rules;
  • display and indexing behavior; and
  • validation and maintenance responsibility.

Profiles turn general standards into accountable local practice. They should not redefine established elements casually. If a local need cannot be represented, a documented local extension may be preferable to silently changing an existing element’s meaning.

4.4.6 Identifiers and access points

An identifier distinguishes an entity within a defined system. Examples include ISBNs, ISSNs, DOIs, ORCID iDs, accession numbers, barcodes, control numbers, handles, and URIs. Identifiers improve matching and exchange only when their scope is understood.

A string can look unique without being persistent. File names, shelf locations, and database row numbers may change. Projects should document the identifier’s assigning authority, entity, uniqueness, persistence expectation, display form, and resolution behavior.

An access point is a name, term, code, or other representation through which a user can retrieve a record. An authorized access point provides a consistent form; variant access points preserve other forms and lead users to the represented entity.

4.4.7 Authority control

Authority control manages identities, preferred forms, variants, sources, and relationships. It is not simply forcing every name into one spelling. A useful authority record may connect:

  • an authorized form;
  • variant names, earlier names, pseudonyms, transliterations, and local forms;
  • identifiers from trusted systems;
  • dates or contextual attributes needed for distinction;
  • relationships to people, families, organizations, places, and works;
  • sources consulted and evidence for decisions; and
  • notes about scope, uncertainty, or community preference.

Figure 4.2 illustrates why authority work is relational. Different labels can point to one identity, while one ambiguous label can refer to different identities.

As Figure 4.2 shows, authority data support collocation and exploration. They also prepare repositories for linked data (Wolverton and Jin 2013). However, an authority file is not infallible. Identities can be uncertain, names can be contested, and communities may prefer forms different from legacy sources. Records need provenance and revision pathways.

4.4.8 Subject access and classification

Subject access attempts to represent what a resource is about, what form it takes, and sometimes its geographic, chronological, or demographic context. Tools include subject-heading lists, thesauri, taxonomies, classifications, genre/form vocabularies, and locally maintained terms.

A controlled vocabulary improves consistency by managing preferred terms, synonyms, broader and narrower concepts, and related concepts. A classification scheme places resources into a systematic arrangement, often supporting shelf order and browsing. Neither is neutral. Categories emerge from particular histories, institutions, languages, and assumptions.

Good subject analysis requires evidence from the resource and appropriate contextual sources. Cataloguers should distinguish what a resource is about from whom it is by, where it is stored, or who is expected to use it. Exhaustivity and specificity should match project policy. Overly broad terms hide distinctions; excessive unsupported specificity creates false precision.

Local and community vocabularies may represent concepts absent from larger systems. Mapping can support interoperability, but a local concept should not be erased merely because no exact external equivalent exists. The mapping may be close, broader, narrower, related, or intentionally absent.

4.4.10 Metadata quality

Quality means fitness for stated purposes and communities, not maximal fullness. A record can be complete according to a template yet inaccurate, inaccessible, or harmful. Research on repository metadata repeatedly identifies consistency, completeness, accuracy, provenance, and interoperability as recurring quality concerns (Park 2009).

Table 4.5 provides operational questions. Quality dimensions can conflict: adding detail may improve discovery but create privacy risk; using one controlled vocabulary may improve consistency but weaken community representation.

As Table 4.5 indicates, a quality program needs evidence. Validation catches some structural errors but cannot determine whether a culturally sensitive description is appropriate. Automated checks and human review should complement each other.

4.4.11 Interoperability, crosswalks, and loss

Metadata interoperability includes technical, syntactic, semantic, and organizational dimensions. Two systems can exchange XML while disagreeing about what “creator” or “date” means. Shared syntax does not guarantee shared semantics.

A crosswalk maps elements between schemas. Crosswalks support migration and aggregation but are rarely lossless. A detailed MARC statement with roles, indicators, and controlled subfields may map to one broad Dublin Core creator value. When mapping many source elements into one target element, reconstructing the original distinctions may be impossible.

Before conversion, teams should:

  1. document source and target definitions;
  2. identify one-to-one, one-to-many, many-to-one, and unmappable cases;
  3. preserve original records or exports;
  4. test representative difficult records;
  5. record transformation rules and software versions;
  6. measure information loss and validation errors; and
  7. obtain review from metadata specialists and affected communities.

Interoperability should not mean flattening every local distinction. Aggregators can retain local values and provide mapped access values with provenance.

4.4.12 Workflow, provenance, and maintenance

Metadata is created through workflows: selection, examination, transcription, normalization, authority work, subject analysis, rights review, encoding, validation, publication, correction, and migration. Each handoff should have an accountable role.

Record provenance should identify sources, creator or agent, creation and modification dates, imported versus locally supplied statements, automated transformations, review status, and uncertainty. Batch-generated or AI-suggested metadata should be marked and sampled for error.

Maintenance is essential. URLs break, organizations change names, rights expire, vocabularies revise terms, and users identify errors. A correction mechanism should allow evidence-based revision without silently erasing record history.

Reflective prompt. Find a catalogue or repository record that helped or hindered you. Which element, value, relationship, or missing context produced that experience?

4.5 Philippine and Southeast Asian examples

4.5.1 University theses: names, departments, and advisers

A Philippine university repository receives theses from many colleges. Department names change over time, students enter advisers in free text, and Filipino names appear in inconsistent order. The repository creates authority identifiers for people and organizations, retains historical department names with dates, and records adviser roles separately from authorship. This improves browse lists and prevents organizational history from being overwritten.

4.5.2 Municipal photographs: supplied titles and local knowledge

A city library digitizes photographs with captions written decades after the events. Staff distinguish transcribed captions from supplied titles and community-contributed identifications. Each statement records source and date. Community corrections are reviewed rather than inserted anonymously, preserving both improved access and provenance.

4.5.3 Indigenous cultural heritage: negotiated access metadata

A cultural institution holds recordings connected to an Indigenous community. Generic “open” or “closed” fields cannot express community protocols. Representatives participate in terminology, sensitivity review, access categories, and correction procedures. Public discovery metadata may remain visible while files have culturally governed conditions. Local values are documented and mapped only where appropriate.

4.5.4 Government records: office names across reorganizations

A government information service describes reports issued by agencies that have merged or changed names. Authority records connect earlier and later corporate bodies without pretending they are identical. Issuing body, current custodian, and subject are recorded separately. Users can search the historical name while navigating to related bodies.

4.5.5 Southeast Asian multilingual collections

A regional portal aggregates records in Filipino, Bahasa Indonesia, Malay, Thai, Vietnamese, and English. It preserves original scripts, adds language tags, stores transliterations as variant access points, and avoids treating English as the authoritative original. Shared type and rights vocabularies support regional filtering while local descriptions retain their source and context.

4.6 Hands-on activity: Describe and review a local heritage item

4.6.1 Purpose

Create a beginner Dublin Core record, document the rules behind its values, and improve it through peer quality review.

4.6.2 Tools

  • One instructor-provided digitized photograph, poster, letter, oral-history excerpt, or pamphlet
  • A spreadsheet or CSV template with metadata fields
  • DCMI Metadata Terms (Dublin Core Metadata Initiative 2020)
  • RightsStatements.org statements (RightsStatements.org 2026)
  • Appropriate name, geographic, language, type, and subject vocabularies
  • A shared class data dictionary

4.6.3 Steps

  1. Examine the object and distinguish visible evidence from supplied contextual information.
  2. State the entity being described: original object, digital surrogate, or both through separate records.
  3. Complete title, creator, contributor, date, type, format, identifier, source, language, subject, spatial coverage, description, and rights fields.
  4. Mark a supplied title according to class policy.
  5. Use at least one controlled name or subject value and record its source.
  6. Use a stable identifier for the item and a URI for the rights statement where applicable.
  7. Record unknown, uncertain, approximate, and not-applicable values according to the profile; do not invent information.
  8. Add record provenance: metadata creator, date created, sources consulted, and review status.
  9. Exchange the record with a peer. The reviewer checks the object, profile, vocabularies, and quality dimensions.
  10. Revise the record and maintain a short change log explaining accepted and rejected suggestions.

4.6.4 Expected output

Submit:

  • one completed metadata record;
  • a data dictionary excerpt for five selected elements;
  • authority or vocabulary evidence for at least three values;
  • a peer-review checklist;
  • a change log;
  • a short rights and sensitivity note; and
  • a 200-word reflection.

Reflection question. Which metadata decision required transcription, which required a standard, which required community or contextual knowledge, and which remained uncertain?

4.7 Ethical and professional issue: description, power, and accountable revision

Description is an exercise of power. Cataloguers decide which names become prominent, which relationships are recorded, which terms group resources, and which histories remain invisible. Legacy records may contain colonial language, racial or ethnic slurs, medicalized terminology, imposed names, gender assumptions, or descriptions centered on collectors rather than represented communities.

Removing harmful language immediately may improve access for current users, but silently replacing every historical statement can erase evidence about institutional practice. A responsible strategy may preserve the original statement in restricted administrative provenance while presenting a revised public value and an explanatory note. The appropriate choice depends on purpose, harm, law, community expectations, and archival integrity.

Community participation should be more than asking people to approve a nearly finished record. Communities may help define entities, preferred names, protocols, access categories, and correction processes. Participation requires time, compensation where appropriate, consent, and clarity about who has final authority.

Rights metadata also has ethical consequences. “Copyright unknown” does not mean “free to use,” while “copyrighted” does not automatically mean inaccessible. Standardized rights statements can communicate known conditions consistently (RightsStatements.org 2026), but staff must document the evidence and jurisdiction behind the choice.

AI systems can suggest captions, subjects, names, translations, or summaries. They can also hallucinate identities, repeat bias, flatten cultural distinctions, and expose restricted content to external services. AI-generated metadata should never be treated as authoritative merely because it is fluent. Projects need approved tools, data-protection review, provenance, confidence thresholds, representative sampling, human validation, and a path for correction.

Professional neutrality does not mean avoiding responsibility. It means applying transparent principles, evidence, community knowledge, intellectual freedom, privacy, and accountable revision rather than personal preference or vendor defaults.

Ethical checkpoint. A legacy photograph caption uses a derogatory term but is the only evidence of how the institution described the community in 1950. What should remain, what should display publicly, who should be consulted, and how should the revision be documented?

4.8 Review questions

  1. Identification: Define metadata element, value, schema, and application profile.
  2. Identification: Distinguish descriptive, structural, technical, rights, provenance, and preservation metadata.
  3. Short answer: Why are MARC 21 and RDA not the same kind of standard?
  4. Short answer: Explain the difference among Work, Expression, Manifestation, and Item.
  5. Short answer: What are the purposes of authorized and variant access points?
  6. Applied analysis: A repository uses one field called date for creation, publication, upload, and copyright. Diagnose the semantic problem and propose a profile revision.
  7. Applied analysis: A project maps detailed MARC records to simple Dublin Core. What information loss should be tested and documented?
  8. Scenario: A community member disputes a subject heading applied to sacred material. Outline an evidence-based and participatory response.
  9. Scenario: An AI tool identifies people in historical photographs. What technical, ethical, privacy, and provenance controls are required before any value is published?
  10. Synthesis: Explain how database design, metadata standards, authority work, and interface display jointly affect discovery.

4.9 Mini case: The island heritage collection

A university library receives 2,000 digitized photographs, field notebooks, and audio recordings documenting an island community from 1948 to 1985. The donor spreadsheet has columns called Name, Date, Place, Tribe, Notes, and Copyright. Some rows mix photographers and depicted persons. Dates range from exact days to guesses. Place names use colonial and current forms. The term in Tribe is rejected by community representatives. Several recordings contain ceremonies not intended for unrestricted circulation. The repository platform offers simple Dublin Core and an automatic image-captioning plugin. The grant requires public launch in four months.

Guide questions

  1. Design an application profile identifying entities, required elements, repeatability, vocabularies, uncertainty, provenance, and access conditions.
  2. Propose a workflow for community participation, rights review, automated suggestions, quality control, correction, and phased release.
  3. Identify what should not be mapped into one public field, and explain how original donor data can be preserved without controlling public description.

4.10 Further reading and tools

4.10.1 Standards and models

  • NISO, Understanding Metadata. A concise introduction to metadata functions and components (Understanding Metadata 2004).
  • DCMI Metadata Terms. Definitions and usage for Dublin Core properties, classes, and vocabularies (Dublin Core Metadata Initiative 2020).
  • MARC 21 Format for Bibliographic Data. Tags, indicators, subfields, and coded structures (Library of Congress 2024b).
  • IFLA Library Reference Model. Entities, relationships, and bibliographic user tasks (IFLA Library Reference Model Review Group 2017).
  • RDA Registry. Open machine-readable RDA elements and vocabularies (RDA Steering Committee 2026).
  • BIBFRAME 2.0 Model. Library of Congress linked-data model for bibliographic description (Library of Congress 2024a).
  • RightsStatements.org. Standardized rights statements for cultural heritage institutions (RightsStatements.org 2026).

4.10.2 Open and widely accessible tools

  • OpenRefine: clean, cluster, reconcile, and transform metadata with an operation history.
  • MarcEdit: validate, transform, and edit MARC and related metadata formats.
  • Zotero: inspect bibliographic metadata and identifiers from common sources.
  • Library of Congress Linked Data Service: browse and retrieve authority and vocabulary data.
  • Getty Vocabularies: reference resources for art, architecture, people, and places.
  • Wikidata: a multilingual linked-data source that requires normal source evaluation.
  • CSVLint and XML validators: check structural conformance of exchange files.
