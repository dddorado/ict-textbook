2  Networking, Web Basics, and Library Websites

2.1 Chapter overview

A library website may be open twenty-four hours a day, but that does not mean every user can reach or use it. A student taps a database link on a phone. The page takes too long to load, redirects to an unfamiliar login screen, and reports that the session has expired. A parent searches a public-library site for today’s opening hours but finds an outdated poster. A researcher follows a repository link from a search engine and receives a “404 Not Found” response. These are not merely website inconveniences. They are failures in the delivery of information services.

The central problem of this chapter is how to make a library service reliably reachable and understandable across networks, browsers, devices, and user contexts. Solving that problem requires more than visual design. It requires a working model of networks, internet addressing, the Domain Name System (DNS), URLs, HTTP and HTTPS, servers, browsers, HTML, CSS, content management, accessibility, performance, and website governance.

Chapter 1 described an information service as a layered sociotechnical system. This chapter opens two of those layers: network infrastructure and the web interface. You will trace a request from a browser to a library server and back, then create a small semantic web page. The aim is not to train you as a network engineer or professional web developer. It is to give you enough technical understanding to diagnose service problems, communicate requirements, maintain trustworthy content, and collaborate effectively with IT staff, vendors, designers, and users.

2.2 Learning objectives

After completing this chapter, you should be able to:

  1. Distinguish the internet, the World Wide Web, local networks, and web-based library services.
  2. Diagram a client-server exchange involving a device, network, DNS, server, HTTP request, and response.
  3. Interpret the principal parts of a URL and common HTTP status codes.
  4. Explain how HTML, CSS, and JavaScript contribute different functions to a web page.
  5. Create a small semantic HTML page with headings, links, lists, contact information, and basic accessibility features.
  6. Evaluate a library website for findability, security, accessibility, mobile use, low-bandwidth performance, and governance.

Table 2.1 introduces the vocabulary used throughout the chapter. Several concepts operate at different layers. DNS helps a client locate a server, while HTML gives structure and meaning to the document returned by that server.

2.4 Core discussion

2.4.1 Networks as service infrastructure

A network exists when devices exchange data according to agreed rules. In a small library, a local area network may connect staff computers, printers, self-check stations, wireless access points, an on-site server, and a router. A university may connect several campus libraries through a larger institutional network. Remote branches, cloud systems, and vendor platforms depend on wide area and internet connections.

Networks allow information organizations to share catalogues, storage, authentication, printers, internet access, and applications. They also create dependencies. If the local network fails, a cloud-hosted circulation system may be unreachable even though the vendor server operates normally. If the internet connection works but DNS fails, a service may be reachable by numerical address while its familiar name appears unavailable. Diagnosis begins by locating the layer at which communication stopped.

Network data are divided into packets, transmitted through links and routing devices, and reassembled at the destination. Two measures help explain user experience:

  • Bandwidth concerns how much data a connection can carry. Large images and video load slowly on a low-bandwidth link.
  • Latency concerns delay. Even a small page can feel slow when it requires many round trips over a high-latency connection.

Bandwidth and latency are not interchangeable. Website teams should measure whether users can complete real tasks rather than treating one advertised connection speed as a full description of service quality.

2.4.2 Internet and web are not synonyms

The internet is the global network infrastructure. The World Wide Web is one service that uses it. Email, file transfer, voice communication, time synchronization, and many APIs also use the internet but are not identical to the web.

The distinction matters. A library can have a working internet connection while its website is unavailable. A catalogue hosted inside a building may work on the LAN during an internet outage but be unreachable from home. An offline exhibit installed on a kiosk can use web technologies without being published on the internet. Accurate language helps teams identify the actual service boundary.

2.4.3 Protocols and layers

A protocol is an agreed set of communication rules. At a simplified level, IP supports addressing and routing; TCP supports a reliable ordered connection for many applications; DNS associates human-readable names with network information; HTTP defines web requests and responses; and TLS protects data in transit for HTTPS.

Students do not need to memorize the full protocol stack. The useful habit is to ask which protocol is responsible for which task. A DNS problem, an expired TLS certificate, and an HTTP “Not Found” response can all block access, but they require different remedies and responsible staff.

2.4.4 From a URL to a page

Figure 2.1 traces a simplified request for a library-hours page. The browser interprets the URL, asks DNS for an address, connects through network infrastructure, establishes a protected session for HTTPS, sends an HTTP request, receives a response, and interprets the returned HTML and related resources.

Figure 2.1 reveals that the browser is an active client. It sends headers, manages cache entries and cookies, validates secure connections, constructs a document model, applies styles, and may run scripts. Browser differences, extensions, privacy settings, and older devices can therefore affect service behavior.

2.4.5 IP addresses, domain names, and DNS

An IP address identifies a network interface for communication and routing. People remember names more easily than numerical addresses, so services use names such as library.example.edu.

DNS is a distributed hierarchical naming system rather than one universal address book. Its foundational design separates the name space from the many servers that answer queries about portions of it (Mockapetris 1987). DNS information is cached for a defined period, so changes may not appear everywhere immediately.

Domain governance is an LIS concern because names support continuity, trust, and citation. An institution should know who controls registration, who manages DNS records, when registrations and certificates expire, how changes are authorized, and which services depend on each name. A public service should not depend permanently on a domain registered under one employee’s private account.

2.4.6 Reading a URL

Consider:

https://library.example.edu/research/guides.html?subject=history#databases

Its scheme is https; its host is library.example.edu; its path is /research/guides.html; its query is subject=history; and its fragment is databases. URLs appear in catalogues, citations, QR codes, syllabi, and archival records. A stable URL policy therefore supports discovery and continuity.

Changing a URL without a redirect breaks inbound links. A migration should inventory important URLs, map old addresses to new ones, configure permanent redirects, and monitor “Not Found” reports. Human-readable paths help users judge purpose, but appearance alone is not proof of safety. Users should inspect the complete host name before entering credentials.

2.4.7 HTTP requests and responses

HTTP is a request-response protocol. RFC 9110 defines current HTTP semantics for methods, status codes, fields, and representations (Fielding et al. 2022). A request includes a method, target, headers, and sometimes a body. A response includes a status code, headers, and often HTML, JSON, an image, or a PDF.

Common methods include GET for retrieving a representation, HEAD for retrieving response metadata, POST for submitting data, and PUT or PATCH for creating or modifying resources in systems that support them.

Table 2.2 lists status codes that LIS staff may encounter. A status is evidence, not a complete diagnosis. A 404 response says that the server found no current resource at that path; it does not explain whether the resource was removed, renamed, mistyped, or migrated without a redirect.

2.4.8 HTTPS, certificates, and trust

HTTPS protects HTTP traffic with TLS. Correct configuration supports confidentiality, integrity in transit, and server authentication. It does not prove that content is accurate or ethical, and it does not protect data after they reach an insecure server. It secures an important channel, not the entire service.

Certificate warnings may indicate expiration, a host-name mismatch, incomplete configuration, or interception. Libraries should monitor certificates and renew them before expiration. Staff should never make bypassing a warning the routine solution.

2.4.9 HTML, CSS, and JavaScript

The browser commonly receives HTML, then requests style sheets, scripts, images, and fonts. Figure 2.2 separates three technologies that beginners often confuse.

HTML identifies headings, paragraphs, navigation, lists, forms, tables, and images. CSS controls presentation and responsive layout. JavaScript adds behavior such as menu expansion, form validation, and asynchronous updates. The WHATWG standard defines HTML as a language for structuring web documents and applications (WHATWG 2026).

As Figure 2.2 shows, technology does not replace content governance. A large bold paragraph may look like a heading but is not programmatically identified as one. A clickable generic container styled like a button lacks the native semantics and behavior of a real button. Semantic HTML helps browsers, screen readers, search engines, and maintenance tools interpret meaning.

2.4.10 A minimal semantic library page

This example publishes an essential service with meaningful structure:

<!doctype html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Library Hours | Mabini Community Library</title>
</head>
<body>
  <header>
    <h1>Mabini Community Library</h1>
    <nav aria-label="Primary">
      <a href="index.html">Home</a>
      <a href="catalogue.html">Catalogue</a>
      <a href="contact.html">Contact</a>
    </nav>
  </header>
  <main>
    <h2>Library hours</h2>
    <p>Monday-Friday: 8:00 a.m.-5:00 p.m.</p>
    <p>Saturday: 9:00 a.m.-12:00 noon</p>
    <h2>Before you visit</h2>
    <ul>
      <li>Bring an ID for registration.</li>
      <li>Contact us before requesting archival materials.</li>
    </ul>
  </main>
  <footer>
    <address>Email: <a href="mailto:library@example.org">library@example.org</a></address>
    <p>Last reviewed: 18 June 2026</p>
  </footer>
</body>
</html>

The language attribute supports language-aware tools. The character encoding supports consistent display. The viewport instruction supports mobile layouts. The title distinguishes the page in browser tabs and search results. Headings communicate organization, link text names destinations, and the review date supports content governance.

2.4.11 Progressive enhancement and resilience

Progressive enhancement begins with meaningful content and basic functions that work under varied conditions, then adds richer presentation and behavior. Opening hours should remain readable if a script fails. A catalogue destination should be a real link. A form should retain understandable labels when visual styling is unavailable.

This approach does not prohibit modern applications. It asks teams to identify the essential service and protect it from avoidable dependencies. Resilience matters for older devices, assistive technologies, intermittent signals, data-saving settings, and restricted institutional computers.

2.4.12 Library websites as service gateways

A library website is not an online brochure. It connects users to catalogues, databases, repositories, guides, reference help, policies, events, spaces, accounts, and staff expertise. Its information architecture should reflect user tasks rather than the institution’s internal chart.

Common tasks include checking hours, finding a known item, searching a topic, accessing electronic resources, renewing materials, asking for help, learning access rules, and verifying whether information is current and official.

Library usability research demonstrates the value of observing users rather than assuming professional terms are self-explanatory (Emde et al. 2009). Accessibility research similarly shows that meaningful access requires continuing attention rather than a one-time launch check (Billingham 2014). Recent homepage work still connects structural and visual decisions with usability and accessibility (Vargas 2026).

2.4.13 Content management and hosting

A content management system may provide templates, navigation, revision histories, user roles, scheduled publication, and approval workflows. It can improve distributed publishing, but it creates governance and security work. Plugins require updates, accounts require review, themes may introduce accessibility problems, and old pages may remain searchable. The institution still needs content owners, review schedules, backup procedures, and an exit plan.

Hosting may use institutional servers, a managed provider, cloud infrastructure, or a hosted software service. The choice affects control, expertise, security updates, cost, performance, jurisdiction, backup, export, and vendor dependence. “In the cloud” does not remove institutional responsibility.

Remote access may involve a proxy, VPN, or federated identity service. Each transition should tell users where they are going and why authentication is required. Repeated redirects and unclear domains can make legitimate access resemble phishing. Libraries should document operators, data exchange, support responsibility, and what happens when a subscription ends.

2.4.14 Accessibility, mobile use, and performance

WCAG 2.2 requires web content to be perceivable, operable, understandable, and robust (World Wide Web Consortium 2023). Mobile accessibility is not separate: small screens, touch input, orientation, magnification, and device constraints interact with the same principles (World Wide Web Consortium 2015).

Beginner checks include meaningful titles, logical headings, text alternatives, form labels, keyboard operation, adequate contrast, text reflow, clear link purpose, and understandable errors. Accessibility benefits many users, but that broad benefit should not erase the specific rights and expertise of persons with disabilities.

Performance is also an access issue. Large posters, multiple fonts, trackers, and third-party widgets consume data and add failure points. Regional digital plans recognize that meaningful participation depends on both infrastructure and capability (Association of Southeast Asian Nations 2021). Practical measures include compressing images, avoiding autoplay, limiting external scripts, offering text alternatives, making essential information available without accounts, and testing on a low-cost phone and constrained connection.

2.4.15 Website governance

Library websites are published information resources and organizational records. Table 2.3 connects components to accountable questions. No component is purely technical: each has a service owner, policy implication, and maintenance need.

As Table 2.3 suggests, publishing is a lifecycle. A page needs a purpose, audience, owner, approval status, review date, and eventual disposition. Event notices expire quickly; policy pages need revision histories; research guides require subject review; and significant exhibits may warrant preservation.

Reflective prompt. Recall a website that prevented you from completing a task. Which layer failed: connection, naming, security, response, page structure, content, accessibility, or governance? What evidence supports your diagnosis?

2.5 Philippine and Southeast Asian examples

2.5.1 State university library: the remote-access chain

A Philippine state university links subscribed databases from its library website. On campus, access is recognized by network address. Off campus, students authenticate through a proxy. Success depends on current vendor links, DNS, proxy configuration, identity records, HTTPS certificates, license rules, and help documentation. When a vendor changes its domain, old proxy links fail. The library responds with a link inventory, off-campus testing, clear escalation information, and coordinated updates.

2.5.2 Municipal library: hours during a typhoon

A municipal library normally posts hours as a colorful image. During a typhoon, staff update social media but forget the website image and PDF. Residents see three schedules. The library redesigns the page so current status appears as lightweight HTML with an update time. One role approves emergency updates, and expired notices are removed through a documented workflow.

2.5.3 School library: service during internet outages

A rural school library uses a local server to provide reading materials over campus Wi-Fi. The collection remains available inside the school when the external connection fails. Staff distinguish the local web from the internet, maintain offline backups, and use a text-first interface suitable for older phones.

2.5.4 Regional heritage portal: multilingual stable links

A Southeast Asian heritage consortium aggregates collections from several museums. Partners use different scripts, languages, domains, and systems. They adopt Unicode, language metadata, institutional identifiers, persistent item URLs, and redirects. The interface presents multilingual values and provenance rather than replacing local description with one regional label.

2.5.5 Public information service: external maps and privacy

A public library embeds a commercial map and analytics on every page. The map helps some visitors but adds tracking and a failure dependency. The library provides its address, landmarks, transport directions, and telephone number as HTML before the map. It then evaluates whether analytics can collect less data.

2.6 Hands-on activity: Trace and build a library web page

2.6.1 Purpose

Trace a real web request, then create a semantic HTML page that communicates an essential library service under mobile and low-bandwidth conditions.

2.6.2 Tools

  • A browser with developer tools, such as Firefox, Chrome, or Edge
  • A text editor such as Notepad, Visual Studio Code, or Positron
  • The W3C Nu HTML Checker at https://validator.w3.org/nu/
  • A folder named chapter2-lab

2.6.3 Steps

  1. Open a library, archive, museum, or repository page that requires no personal information.
  2. Record its URL and label the scheme, host, path, query, and fragment where present.
  3. Open the browser Network panel, reload, and select the main document request.
  4. Record the method, status, content type, transferred size, and domain.
  5. Count contacted domains and distinguish institutional from external services.
  6. Use a narrow mobile viewport to find hours, the catalogue, or contact help.
  7. Create hours.html with a document type, language, encoding, viewport metadata, and unique title.
  8. Add header, navigation, main, and footer landmarks with one h1 and logical h2 headings.
  9. Add fictional hours, a status message, visit requirements, contacts, and a review date.
  10. Add three meaningful links. If using an image, resize it and supply useful alternative text.
  11. Test two viewport widths, validate the HTML, and correct errors you understand.
  12. Ask a peer to locate Saturday hours and a contact method without explanation.

2.6.4 Expected output

Submit the annotated request record, three network and mobile observations, hours.html, validation evidence, the peer-task result, and a 150-word reflection.

Reflection question. Which improvement came from understanding the network, which came from HTML structure, and which came from observing a user?

2.7 Ethical and professional issue: access, surveillance, and secure delivery

Library websites can broaden access while excluding, observing, or endangering users. An online-only form may disadvantage people without stable connectivity, compatible devices, data allowance, accessible design, or required identification. The IFLA-UNESCO Public Library Manifesto connects digital access with inclusion and equitable service (International Federation of Library Associations and Institutions and UNESCO 2022). Digital channels should expand options rather than silently eliminate necessary alternatives.

Web requests can disclose IP addresses, device information, referring pages, cookies, and behavioral patterns. Third-party fonts, maps, videos, chat widgets, and social-media embeds may transmit data before intentional interaction. Before adding one, ask what is sent, to whom, for what purpose, for how long, and whether the essential service works without it.

The Philippine Data Privacy Act’s principles of transparency, legitimate purpose, and proportionality apply to forms, logs, analytics, and vendor integrations (Republic of the Philippines 2012; National Privacy Commission 2023). A notice cannot by itself legitimize unnecessary collection.

HTTPS should be standard, especially for authentication, forms, and accounts. Interface choices also influence security. A database link that silently opens an unfamiliar domain and asks for credentials may train users to ignore phishing indicators. The page should name the provider or authentication step, use consistent domains where possible, and offer a trusted help route.

Security messages should support rather than shame users. “Your session expired. Sign in again through the library portal” is more useful than “Invalid error.” Good error design helps distinguish a normal interruption from a suspicious request.

Ethical checkpoint. A free chat widget permits evening reference service, but its vendor stores transcripts and device identifiers in another jurisdiction. What benefit, data flow, contract term, alternative, and user notice should be examined?

2.8 Review questions

  1. Identification: What is the difference between the internet and the World Wide Web?
  2. Identification: Define bandwidth and latency. How can each affect a library website?
  3. Short answer: What role does DNS play after a user enters a domain name?
  4. Short answer: Identify the URL parts in https://archive.example.ph/items/42?view=full#description.
  5. Short answer: Distinguish the roles of HTML, CSS, and JavaScript.
  6. Applied analysis: A catalogue works on staff computers but not from homes. Give three plausible causes from different layers and one test for each.
  7. Applied analysis: After a repository migration, search results lead to 404 responses. What should the team have prepared, and what should it do now?
  8. Scenario: A homepage loads 12 MB of poster images before showing hours. Evaluate access, performance, metadata, and maintenance.
  9. Scenario: A browser reports an invalid certificate. Why should staff not tell users to bypass the warning?
  10. Synthesis: Choose one component from Table 2.3 and explain how its maintenance connects to an LIS responsibility.

2.9 Mini case: The multicampus library website

Eastern Bay University has three campuses and a growing distance-learning program. Each library maintains separate CMS pages. Labels differ: one says “OPAC,” another “Books,” and the third “Library Search.” Hours are poster images, two from last semester. Database links work on the main campus but fail for many remote students. The website loads six tracking and social-media domains. A screen-reader user reports repeated unlabeled links. Communications controls templates, IT controls authentication and DNS, vendors control database platforms, and no office believes it owns the complete journey.

Guide questions

  1. Map the technical and organizational components from homepage to authenticated database article. Where should evidence be collected?
  2. Prioritize five improvements using impact, urgency, effort, accessibility, privacy, and institutional control.
  3. Propose governance naming content owners, technical owners, review intervals, escalation paths, and two success measures.

2.10 Further reading and tools

2.10.1 Standards and documentation

  • RFC 1034, Domain Names—Concepts and Facilities. Foundational DNS concepts (Mockapetris 1987).
  • RFC 9110, HTTP Semantics. HTTP methods, status codes, fields, and representations (Fielding et al. 2022).
  • WHATWG HTML Living Standard. The normative reference for HTML semantics (WHATWG 2026).
  • WCAG 2.2. Testable web accessibility requirements (World Wide Web Consortium 2023).
  • W3C mobile accessibility guidance. Application of accessibility guidance to mobile contexts (World Wide Web Consortium 2015).

2.10.2 LIS readings

  • Emde et al. (2009) demonstrates task-based testing with academic-library users.
  • Billingham (2014) discusses improving academic-library access for people with disabilities.
  • Vargas (2026) connects homepage structure and aesthetics with usability and accessibility.

2.10.3 Open and widely accessible tools

  • Browser developer tools: inspect requests, status codes, structure, responsive layouts, and accessibility.
  • W3C Nu HTML Checker: identify HTML conformance errors and warnings.
  • WAVE: support human accessibility review by marking potential issues.
  • Lighthouse: generate performance and accessibility diagnostics.
  • WebPageTest: examine loading under varied devices and connections.
  • diagrams.net: draw request paths and responsibility maps.
  • Visual Studio Code or Positron: edit HTML with syntax support.
