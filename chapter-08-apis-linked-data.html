8  APIs, Linked Data, and the Semantic Web

8.1 Chapter overview

A reference librarian maintains a subject guide on Philippine children’s literature. Every month, the librarian searches several catalogues, copies titles and authors into a table, checks cover links, and updates availability notes. The guide is useful, but the process is slow and quickly becomes outdated. A public books API could supply structured records, yet its creator names, subjects, editions, and cover images do not always match local cataloguing practice. A linked-data service could connect authors and subjects to stable identifiers, but those links may also import errors or externally imposed descriptions.

The central problem of this chapter is how information systems exchange structured data and meaningful relationships without losing provenance, context, governance, or professional judgment. An application programming interface (API) lets one software system request functions or data from another under a documented contract. Linked data uses web identifiers and explicit statements to connect entities across datasets. The semantic web is the broader vision of web data whose meanings and relationships can be processed across organizational boundaries.

These technologies matter throughout LIS. A catalogue retrieves cover images and availability. A repository exposes metadata for harvesting. A digital exhibit embeds manifests. A reference guide draws from an authority service. A records dashboard combines data from agencies. A discovery layer queries indexes and link resolvers. An archives portal connects people, places, organizations, collections, and events. The technical exchange is only one layer: terms of use, privacy, attribution, rate limits, data quality, sustainability, and community authority determine whether reuse is responsible.

This chapter introduces API requests and responses, endpoints, parameters, authentication, HTTP methods and status codes, JSON and XML, pagination, rate limiting, error handling, identifiers, RDF triples, vocabularies, ontologies, knowledge graphs, JSON-LD, and SPARQL. R examples use httr2 and jsonlite, but the concepts apply across programming languages and platforms (Wickham 2026; Ooms 2026).

8.2 Learning objectives

After completing this chapter, you should be able to:

  1. Describe an API exchange using client, endpoint, method, parameters, headers, status, and response body.
  2. Interpret nested JSON and XML structures and select fields needed for an LIS service.
  3. Construct and document a read-only API request in R with error, pagination, rate-limit, and provenance handling.
  4. Express bibliographic or cultural-heritage relationships as RDF subject-predicate-object triples using stable identifiers.
  5. Read and modify a simple SPARQL SELECT query.
  6. Evaluate data reuse for licensing, privacy, bias, provenance, accessibility, reliability, and sustainability.

8.4 Core discussion

8.4.1 Interfaces as service contracts

An API is not simply “a URL that returns data.” It is a contract describing what may be requested, how requests are expressed, how responses are structured, what errors mean, which uses are authorized, and how service changes are communicated. Documentation may specify endpoints, parameters, examples, schemas, authentication, rate limits, versions, licensing, and support.

In a graphical interface, a person selects a search field and clicks a button. With an API, software expresses the action directly. This enables repeatable integration, but it removes many visual warnings. A programmer must read the documentation and make limits explicit.

8.4.2 Anatomy of a request

Consider this illustrative URL:

https://openlibrary.org/search.json?q=philippine+folktales&fields=key,title,author_name,first_publish_year&limit=20

It contains:

  • the scheme https;
  • the host openlibrary.org;
  • the endpoint path /search.json;
  • query parameters after ?;
  • parameter-value pairs separated by &.

The query value must be URL-encoded. Libraries should use an HTTP client rather than construct complex URLs through manual concatenation. An HTTP client manages encoding, headers, timeouts, and errors more reliably.

HTTP methods have conventional meanings. GET retrieves a representation and should not alter data. POST commonly creates a resource or submits processing. PUT replaces or creates at a known identifier, PATCH applies a partial change, and DELETE requests removal. Actual behavior depends on the API contract. A method that modifies a production catalogue is a consequential action requiring authorization, validation, logging, and usually confirmation.

Table 8.2 lists common status families and reasonable client responses. HTTP semantics are standardized, while each API defines additional error details (Fielding et al. 2022).

As Table 8.2 shows, retrying every failure is unsafe. Repeating a malformed request wastes capacity; repeating a write may create duplicates unless the operation is idempotent.

8.4.3 Authentication, authorization, and secrets

Some public APIs allow anonymous reading. Others require an API key, OAuth token, institutional identity, or user authorization. Authentication establishes identity; authorization determines allowed actions. An API key should not be embedded in a published .qmd, committed to Git, printed in logs, or pasted into a public issue.

Credentials can be stored in environment variables and read at runtime:

api_key <- Sys.getenv("BOOKS_API_KEY")
if (api_key == "") stop("BOOKS_API_KEY is not configured")

Environment variables reduce accidental disclosure but do not replace secure secret storage, rotation, restricted permissions, and incident response. A client should request only the scopes it needs. A guide that reads public book data does not need permission to modify records.

8.4.4 JSON: nested objects and arrays

JSON is a text interchange format built from objects, arrays, strings, numbers, booleans, and null (Bray 2017). Objects use name-value pairs; arrays contain ordered values.

{
  "key": "/works/OL123W",
  "title": "Mga Kuwentong Bayan",
  "authors": [
    {"name": "A. Dela Cruz", "id": "/authors/OL45A"}
  ],
  "first_publish_year": 1998,
  "languages": ["fil", "eng"],
  "open_access": true,
  "description": null
}

The top level is an object. authors is an array of objects; languages is an array of strings; open_access is logical; description is explicitly null. A rectangular table cannot represent every nested relationship without transformation. Flattening authors into one string is convenient but may discard identifiers and roles.

8.4.5 XML and namespaces

XML represents nested elements, attributes, and text (World Wide Web Consortium 2008). Library standards frequently use XML, including MARCXML, Dublin Core exchanges, OAI-PMH responses, and some authority services.

<record xmlns:dc="http://purl.org/dc/elements/1.1/">
  <dc:identifier>local:123</dc:identifier>
  <dc:title xml:lang="fil">Mga Kuwentong Bayan</dc:title>
  <dc:creator>A. Dela Cruz</dc:creator>
  <dc:language>fil</dc:language>
</record>

The dc prefix is bound to a namespace URI. The prefix itself can change while the namespace identity remains. Parsing XML by searching strings such as <dc:title> is fragile because whitespace, prefixes, attributes, and nesting can vary. Use an XML parser that understands namespaces.

JSON and XML are serializations, not guarantees of semantics. A JSON property named author and an XML element named creator may or may not express the same rule. Interoperability requires documentation, schema mapping, and shared or reconciled vocabularies.

8.4.7 Pagination, rate limits, caching, and change

APIs rarely return every matching record in one response. Pagination may use page numbers, offsets, cursors, or continuation tokens. The response may include next, total, or limit information. A client should stop deliberately and detect repeated tokens to avoid an infinite loop.

Rate limits protect shared services. A 429 response often indicates that the client should wait, reduce its request frequency, or consult a response header. Adding many parallel requests can worsen the problem. Caching a permitted response reduces network use and supports reproducibility, but cached data need a retrieval date and refresh policy.

APIs evolve. Fields can be added, renamed, restructured, or deprecated. Versioned endpoints help but do not eliminate change. A sustainable integration requires:

  • schema and sample-response tests;
  • timeouts and bounded retries;
  • logs without secrets or unnecessary personal data;
  • a cached or saved raw response when licensing permits;
  • clear failure display rather than stale data presented as current;
  • an owner who monitors documentation and service notices;
  • a fallback or exit plan.

8.4.8 From identifiers to linked data

Traditional records often store labels: Manila, Jose Rizal, or Children's literature. Labels are ambiguous and vary by language and time. Linked data uses identifiers for things and properties, while retaining labels for people.

The foundational linked-data guidance recommends using HTTP identifiers, providing useful information when identifiers are looked up, and linking to other identifiers (Berners-Lee 2006). An identifier is not automatically correct or permanent, but it provides a reference that systems can distinguish from a text label.

Persistent identifier systems include DOI for digital objects, ORCID for researchers, and locally governed archival or authority identifiers. Persistence is an organizational promise supported by governance and redirection, not a visual property of a URL (International DOI Foundation 2026).

8.4.9 RDF triples: statements as a graph

RDF represents statements as triples: subject, predicate, object (World Wide Web Consortium 2014). The subject is an identified resource. The predicate identifies a property or relationship. The object is another resource or a literal.

<https://example.edu/work/123>
    <http://purl.org/dc/terms/title>
    "Mga Kuwentong Bayan"@fil .

<https://example.edu/work/123>
    <http://purl.org/dc/terms/creator>
    <https://example.edu/person/45> .

The first object is a Filipino-language literal. The second is another identified resource. Using a creator URI allows further statements about the person without repeating all details in every work record.

As Figure 8.2 shows, a graph can connect work, instance, person, and concept while keeping labels and relationships distinct. BIBFRAME applies linked-data principles to bibliographic description and distinguishes conceptual and material aspects such as Work, Instance, and Item (Library of Congress 2024).

8.4.10 Vocabularies, ontologies, and application profiles

A vocabulary supplies identified terms; an ontology adds formal expression of classes, properties, relationships, and sometimes constraints. The difference is not always sharp in practice. DCMI Metadata Terms provides properties and classes useful across domains (Dublin Core Metadata Initiative 2020). BIBFRAME and RDA vocabularies express bibliographic entities and relationships (Library of Congress 2024; RDA Steering Committee 2026).

Reusing a vocabulary does not mean using every term. An application profile identifies which terms a service uses, whether they are required or repeatable, their expected value types, and local constraints. Chapter 4’s distinction among element sets, value vocabularies, content rules, and encodings remains essential.

The Library of Congress Linked Data Service publishes authorities and vocabularies as dereferenceable identifiers (Library of Congress 2026). Linking a local subject to an external authority can improve reconciliation, but the local institution remains responsible for whether that concept represents the community appropriately.

8.4.11 JSON-LD: linked data in JSON

JSON-LD expresses RDF-compatible linked data using JSON syntax and a context that maps short terms to identifiers (World Wide Web Consortium 2020).

{
  "@context": {
    "dcterms": "http://purl.org/dc/terms/",
    "title": "dcterms:title",
    "creator": {"@id": "dcterms:creator", "@type": "@id"}
  },
  "@id": "https://example.edu/work/123",
  "title": {"@value": "Mga Kuwentong Bayan", "@language": "fil"},
  "creator": "https://example.edu/person/45"
}

Without the context, title and creator are merely local JSON property names. The context connects them to DCMI properties and declares that the creator value is an identifier rather than ordinary text. Contexts must be governed and preserved; changing a mapping can change meaning without changing the visible short property name.

8.4.12 SPARQL: asking graph-shaped questions

SPARQL queries RDF graphs by matching triple patterns (World Wide Web Consortium 2013). A basic query uses prefixes, selects variables, and specifies patterns in a WHERE block:

PREFIX dcterms: <http://purl.org/dc/terms/>

SELECT ?work ?title
WHERE {
  ?work dcterms:subject <https://example.edu/concept/folktales> ;
        dcterms:title ?title .
  FILTER(LANG(?title) = "fil")
}
LIMIT 20

The semicolon reuses the same subject. The query asks for works associated with one concept and Filipino-language titles. SPARQL can join across relationships without flattening everything into one table. Public endpoints such as the Wikidata Query Service provide environments for learning, subject to service limits and data quality (Wikimedia Foundation 2026).

SPARQL results are not inherently authoritative. They reflect what is present in the graph, which identifiers were linked, and which modeling decisions were made.

8.5 Philippine and Southeast Asian examples

8.5.1 Academic library: an API-powered subject guide

A Philippine university library builds a guide to disaster-risk-reduction publications. A script queries an open bibliographic API for candidate titles, stores the raw response and retrieval date, and produces a review table. Librarians verify local relevance, editions, creator names, rights, and links before publication. The guide labels external data sources and includes locally catalogued government reports that the API does not cover.

8.5.2 National and local authority work

Libraries and archives may reconcile names of Philippine agencies across reorganizations. A knowledge graph can represent that one agency succeeded another, used an earlier name during a period, and issued particular records. The model should preserve dates, source evidence, and uncertainty. It should not merge organizations only because acronyms match.

8.5.3 Cultural heritage and community identifiers

A regional museum portal links objects to place and community identifiers. External global datasets improve cross-institution discovery but may contain colonial names, disputed boundaries, or categories rejected by communities. The portal preserves local preferred labels, records external matches with provenance and confidence, and allows contributing institutions to reject or revise links.

8.5.4 Public library mobile service

A city library’s mobile page calls an API for branch hours and service alerts. The interface caches the last successful response for low-bandwidth use but clearly displays when it was updated. If the API fails, the page shows a telephone and text-message contact rather than an empty schedule. Operational data are separated from patron account data.

8.5.5 Government open-data integration

A civic information service combines agency datasets. Similar field names such as location, region, and office do not necessarily have the same meaning. The team publishes a data dictionary and transformation log, retains source identifiers, checks licenses, and avoids joining personal records merely because technical identifiers make linkage possible.

Reflect: Name two datasets that could improve a local information service if linked. What identifier would connect them, and what harm could result from an incorrect or excessive link?

8.6 Hands-on activity: Build an API-powered subject-guide dataset

8.6.1 Purpose

Retrieve a small public API response, preserve the raw data, extract selected fields, document provenance and limitations, and produce a review-ready subject-guide table. This lab can complete the database or API mini-project.

8.6.2 Tools

  • R and Quarto;
  • packages httr2, jsonlite, dplyr, readr, and knitr;
  • the official Open Library API documentation (Internet Archive 2026);
  • a topic connected to an LIS service, such as local history, children’s literature, archives, preservation, media literacy, or disaster information.

8.6.3 Steps

  1. Write a one-sentence audience and information need. Define inclusion and exclusion criteria before requesting data.
  2. Read the API documentation. Record endpoint, parameters, fields, authentication, rate limits, pagination, terms, and attribution requirements.
  3. Design a request for no more than 20 results and only necessary fields. Use GET, a descriptive user agent, HTTPS, and a timeout.
  4. Run the request once. Record retrieval time, final URL without secrets, HTTP status, relevant headers, and the source’s documentation URL.
  5. Save the raw JSON response unchanged as data/raw_response_YYYY-MM-DD.json if the terms permit.
  6. Inspect structure before flattening. Note missing fields, arrays, nested objects, identifiers, and nulls.
  7. Extract source key, title, first creator, first publication year, language values, and one source link. Preserve multiple values or document how they were reduced.
  8. Add local fields: selection_status, selection_reason, verified_by, verified_date, and local_note. Do not present API results as endorsed resources before review.
  9. Validate row count, unique source keys, required titles, plausible years, and valid source URLs. Send anomalies to an exception table.
  10. Produce a review table and a separate five-item publishable table. Include source attribution and retrieval date.
  11. Express one selected book, creator, and topic as at least five RDF-like triples. Distinguish URI objects from literals and add a language tag where appropriate.
  12. Render a Quarto report containing method, request documentation, data dictionary, tables, limitations, ethics, and maintenance plan.

8.6.4 Expected output

Submit:

  • a reproducible .qmd report and rendered HTML or DOCX;
  • saved raw response, subject to source terms;
  • cleaned CSV and exception CSV;
  • request and response metadata without credentials;
  • data dictionary and attribution statement;
  • five RDF-like triples with namespace notes;
  • five reviewed subject-guide entries; and
  • a 200-word reflection: What meaning was lost or added when the API response became a local guide?

8.7 Ethical and professional issue: linkage, consent, and contextual integrity

Linked data can make relationships visible, but visibility is not always harmless. Connecting an archival name to a current identity, a historical address to a household, or a community object to a sacred place can expose people or knowledge beyond the context in which information was provided. Public availability in one database does not automatically justify aggregation, enrichment, or republication in another.

APIs also make extraction scalable. A website intended for individual lookup may expose an API that enables bulk profiling. Technical access is not the same as ethical permission. Professionals must examine law, license, consent, expectations, community protocols, and the consequences of combination.

Responsible API and linked-data practice includes:

  • collect only fields necessary for the stated service;
  • distinguish identifiers, assertions, labels, and inferred links;
  • record source, retrieval time, transformation, license, and confidence;
  • retain local or community-preferred description when external links are added;
  • avoid publishing precise sensitive locations or personal relationships without authority;
  • provide correction, unlinking, and takedown routes;
  • test for language, geographic, historical, and collection bias;
  • do not expose API keys, tokens, patron identifiers, or query histories;
  • review vendor reuse, analytics, retention, and cross-border processing;
  • design a fallback when an external API changes or disappears.

The Philippine Data Privacy Act applies when linked datasets contain personal information, including when separate fields become identifying through combination (Republic of the Philippines 2012; National Privacy Commission 2023). Data minimization and purpose limitation should guide both requests and published outputs.

Machine-generated links require particular care. Entity reconciliation may assign a confidence score, but a high score is not evidence of consent or community acceptance. Namesakes, transliteration, changing boundaries, and organizational succession create plausible false matches. Use automated matching to produce candidates, preserve alternatives, and require appropriate review for consequential links.

Professional judgment prompt: A genealogy API can connect digitized school registers to living family trees. What lawful purpose, consent, time limits, access restrictions, accuracy controls, and community consultation would be required before a library enabled the connection?

8.8 Review questions

  1. Identification: Distinguish endpoint, parameter, header, status code, and response body.
  2. Identification: What is the difference between authentication and authorization?
  3. Short answer: Why should a client preserve a raw response and retrieval metadata when source terms permit?
  4. Short answer: Explain why JSON, an API, and RDF are not interchangeable terms.
  5. Code reading: In a JSON record, authors is an array of objects. What information might be lost by keeping only the first author name?
  6. Applied analysis: A service returns 429 with a retry header. What should the client do, and what should it avoid doing?
  7. Applied modeling: Write three RDF-like triples connecting a digitized photograph, its photographer, and a depicted place. Identify resources and literals.
  8. Scenario: A public API changes author_name from a string to an array. How should schema tests, error handling, and maintenance respond?
  9. Scenario: A knowledge graph links two people because their romanized names and birth years match. What additional evidence and review are needed?
  10. Critical analysis: When would bulk export be more appropriate than an API or SPARQL endpoint for an LIS project?

8.9 Mini case: The regional heritage graph

Five Southeast Asian museums create a shared portal linking objects, makers, places, communities, and historical events. A contractor reconciles names against a global knowledge graph and automatically accepts matches above 90% confidence. The portal gains multilingual labels and external links, but it also merges two artists with similar romanized names, assigns present-day nationalities to people who lived before current borders, exposes the location of a sacred site, and replaces a community-preferred object name with a colonial term. Source institutions cannot see which statements came from them, the contractor, or the external graph.

8.9.1 Guide questions

  1. Which failures involve identity, modeling, provenance, privacy, community authority, and workflow governance?
  2. How should the graph represent competing labels, historical jurisdictions, uncertain matches, and statement-level sources?
  3. Design a correction and approval workflow that preserves useful links without treating automated reconciliation as authoritative.

8.10 Further reading and tools

8.10.1 Readings and standards

  • HTTP Semantics. The authoritative specification for methods, statuses, headers, and HTTP behavior (Fielding et al. 2022).
  • RFC 8259. The JSON data interchange specification (Bray 2017).
  • XML 1.0. The W3C recommendation for XML syntax (World Wide Web Consortium 2008).
  • RDF 1.1 Concepts and Abstract Syntax. The core RDF data model (World Wide Web Consortium 2014).
  • SPARQL 1.1 Query Language. The W3C recommendation for querying RDF (World Wide Web Consortium 2013).
  • JSON-LD 1.1. A JSON-based serialization for linked data (World Wide Web Consortium 2020).
  • BIBFRAME 2.0 Model. Library of Congress documentation for a bibliographic linked-data model (Library of Congress 2024).
  • DCMI Metadata Terms. Reusable properties and classes for resource description (Dublin Core Metadata Initiative 2020).

8.10.2 Open or widely accessible tools

  • Open Library APIs: public book and catalogue-related interfaces for bounded learning projects (Internet Archive 2026).
  • Library of Congress Linked Data Service: browsable and machine-readable authority and vocabulary data (Library of Congress 2026).
  • Wikidata Query Service: a public SPARQL learning and exploration environment (Wikimedia Foundation 2026).
  • httr2 and jsonlite: R packages for HTTP requests and JSON processing (Wickham 2026; Ooms 2026).
  • Postman, Insomnia, or Bruno: graphical clients for inspecting API requests and responses; do not save secrets in shared workspaces.
  • RDF playgrounds and validators: useful for checking syntax, but sensitive institutional data should not be pasted into external services.

8.10.3 Closing synthesis

APIs make functions and data addressable across systems. JSON and XML encode responses. Identifiers distinguish resources from labels. RDF expresses statements as graph relationships. Vocabularies contribute shared meaning, JSON-LD carries that meaning in JSON, and SPARQL asks questions across graphs. Each layer solves a different problem.

For LIS professionals, successful integration is not simply receiving a 200 status or producing a dense network visualization. The relevant question is: Can users and maintainers understand where the data came from, what each relationship means, which values were transformed, what the source permits, which people or communities could be harmed, and what happens when the external service changes? Chapter 9 turns from data exchange to the experience of people using these services, emphasizing usability, accessibility, responsive design, and mobile contexts.
